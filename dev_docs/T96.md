# Tsyganenko T96 / T01 Model Investigation

Research notes for evaluating whether to replace the current T89c external field model with
T96 or T01.

---

## 1. Model Overview

### T89c (current implementation)

- **Inputs**: `iopt` (1–7, binned Kp index), `ps` (dipole tilt, rad), `x/y/z` (GSM, Earth radii)
- **Coefficient table**: 30 params × 7 Kp rows — hard-coded array in `src/physics/t89.js`
- **Implementation**: ~254 lines (`extern()` function)
- **Current systems**: ring current, tail current, closure current, Chapman-Ferraro, Birkeland
- **Kp mapping**: `solarWindToKp({ dst, vSw, nSw })` heuristic in `src/physics/solarWind.js`

### T96 (Tsyganenko 1995–96)

- **Inputs**: `Pdyn` (nPa), `Dst` (nT), `ByIMF` (nT), `BzIMF` (nT), `ps`, `x/y/z`
- **No lookup table** — coefficients are embedded constants in analytical formulas
- **5 current systems**: Chapman-Ferraro, symmetric ring current, cross-tail current sheet,
  field-aligned (Birkeland), IMF penetration
- **Fortran source** (T96_geopack_08.f): ~2,800 lines, ~40 subroutines
- **Python reference**: `geopack` library ([tsssss/geopack](https://github.com/tsssss/geopack)) —
  the same source used to port T89c to JavaScript

### T01 (Tsyganenko 2001–02)

- **Inputs**: `Pdyn`, `Dst`, `ByIMF`, `BzIMF`, **`G1`**, **`G2`** (disturbance indices), `ps`, `x/y/z`
- **G1/G2** quantify solar-wind driving history over the preceding storm interval — they cannot
  be derived from instantaneous solar wind values; they require an hourly data accumulation window
- Adds asymmetric (dawn-dusk) ring current and stronger IMF By effects compared to T96
- ~10–15% larger code than T96

## 2. Parameterisation Comparison

| | T89c | T96 | T01 |
|---|---|---|---|
| Primary driver | Kp bin (iopt 1–7) | Pdyn, Dst, By, Bz | Pdyn, Dst, By, Bz, G1, G2 |
| IMF By (dawn-dusk) | Implicit via Kp | **Explicit** | **Explicit + asymmetric ring** |
| IMF Bz | Indirect (via Kp/Dst mapping) | **Direct** | **Direct** |
| Storm history (G1/G2) | No | No | **Yes — needs hourly data** |
| Coefficient table | 30 × 7 rows | Embedded constants | Embedded constants |
| JS inputs already available | All (Kp from solarWindToKp) | Pdyn, Dst, By, Bz — all in `solarWindParams` | All except G1/G2 |

**Integration note**: T96 inputs map directly onto the existing `solarWindParams` object
(`vSw → Pdyn` via `computeDynamicPressure`, `dst`, `imfBz`; and `imfBy` would be a new slider).
The `solarWindToKp()` heuristic would no longer be needed.

---

## 3. Computational Feasibility

Both T96 and T01 are **analytically evaluated per point** — no numerical integration, root-finding,
or iteration. Each call computes a sum of parameterised field components using closed-form formulas.

| | T89c | T96 | T01 |
|---|---|---|---|
| Estimated JS lines | ~300 | ~800–1000 | ~900–1100 |
| Per-point operations | O(1), ~50 ops | O(1), ~200 ops | O(1), ~230 ops |
| Python speed (geopack) | ~1 ms | ~2–3 ms | ~3 ms |
| Estimated browser speed | <0.1 ms | ~0.1–0.5 ms | ~0.1–0.5 ms |
| External data files | None | None | None (G1/G2 is separate) |
| Existing JS port | None (we wrote it) | **None found** | **None found** |

**Performance is not a blocker**: Field lines are traced asynchronously, not per-frame. A
complete rebuild traces ~100 seeds × ~300 RK4 steps × 4 field evaluations = ~120,000 calls.
At 0.1–0.5 ms/call in JavaScript the total rebuild time would be 12–60 seconds — too slow.
In practice JavaScript is ~100–1000× faster than the Python estimate (V8 vs. CPython), so
the actual cost per call is ~1–10 µs, keeping total rebuilds in the 1–3 second range (same
as T89c). The RK4 tracing loop dominates, not the field evaluation itself.

**Porting risk**: The Fortran source (T96_geopack_08.f) contains copy-pasted code blocks
(noted by the geopack Python author). The Python port refactors these but still runs to ~600
lines. A careful JavaScript port following the same patterns as `t89.js` is feasible but
significantly larger in scope.

## 4. What T96 Would Improve

| Visual issue | T89c behaviour | T96 improvement |
|---|---|---|
| IMF By — dawn-dusk twist | None — By is not used | Field twist proportional to By |
| IMF Bz direct effect | Indirect via Kp/Dst heuristic | Direct — Bz enters field formulas |
| Neutral sheet warping | Fixed parametric form | More accurate parametric shape |
| Kp → Dst mapping noise | `solarWindToKp()` approximation | Removed — Dst used directly |

**At the current visualisation resolution** (~8 latitude bands, 8 longitudes), the difference
between T89c and T96 would be noticeable mainly in:

1. Storms with significant IMF By ≠ 0 (field lines twist asymmetrically dawn-to-dusk)
2. Events where the Dst/Kp mapping in `solarWindToKp()` is notably wrong

For a **teaching / exploration** tool, T89c is adequate. For **event-study or research** use,
T96 adds meaningful accuracy — particularly once real historical solar wind data (Phase 4) drives
the model parameters.

## 5. The G1/G2 Problem for T01

G1 and G2 are time-integrals of solar wind forcing over the preceding storm interval. They
**cannot** be computed from instantaneous solar wind values alone. Three options if T01 is pursued:

| Option | Complexity | Accuracy |
|---|---|---|
| Manual sliders (user enters G1/G2 0–10) | Low | None — purely illustrative |
| Rolling window from playback history | Medium | Fair — accumulates during playback |
| Compute from NOAA/OMNI data stream | High | Good — natural fit for Phase 4 |

**Recommendation**: Skip T01 until historical solar wind data (Phase 4 roadmap) is implemented.
T96 gives most of the accuracy improvement without the G1/G2 complication.

## 6. Implementation Path (if proceeding with T96)

### New files

- **`src/physics/t96.js`** — Port from Python geopack `t96.py`
  - Signature: `t96({ pdyn, dst, byImf, bzImf }, ps, x, y, z) → [bx, by, bz]` (nT, GSM)
  - No `iopt` parameter — inputs are physical values directly
  - Estimated ~800–1000 lines
- **`tests/t96.test.js`** — Mirror `tests/t89.test.js` structure with T96-specific checks:
  - N-S antisymmetry at ps=0
  - By-induced dawn-dusk twist (By > 0 → positive By component in tail)
  - Dst scaling (stronger Dst → larger ring current amplitude)
  - Dayside compression decreases with Pdyn
  - No NaN at origin, surface, tail positions

### Modified files

| File | Change |
|---|---|
| `src/physics/solarWind.js` | `computeExternalB` calls `t96()` instead of `t89()`. `solarWindToKp()` kept for fallback/T89 mode. Pass `pdyn` from `computeDynamicPressure()`. |
| `src/main.js` | Add `imfBy: params.imfBy` to `solarWindParams` object in `getSolarWindParams()` |
| `src/ui/controlPanel.js` | Add `imfBy` slider (−20 to +20 nT) in Solar Wind folder. Update presets. |

### Keeping T89 available

Add a model selector dropdown (T89 / T96) in the GUI. T89 remains the default until T96 is
validated against reference outputs. The selector flag routes through `computeExternalB`.

### Validation approach

Cross-validate `t96.js` output against geopack Python for identical (x, y, z, params) inputs.
Aim for < 0.1 nT agreement at representative points (inner magnetosphere, tail, dawn/dusk flanks).

## 7. Recommendation

| Criterion | Assessment |
|---|---|
| Technically feasible? | **Yes** — analytical O(1), no data files, same calling pattern as T89c |
| Python reference available? | **Yes** — geopack `t96.py` (well-tested, matches Fortran) |
| Existing JS port to leverage? | **No** — novel contribution; must port from Python |
| Inputs already plumbed through? | **Yes** — Pdyn, Dst, Bz all in `solarWindParams`; By needs one new slider |
| Visible improvement at current resolution? | **Moderate** — mainly IMF By twist and Dst accuracy |
| Estimated effort | **~1–2 weeks**: `t96.js` (~800 lines) + tests + integration |
| Blocking issues? | None for T96. T01 blocked on G1/G2 until Phase 4 historical data. |

**Go / No-go**: **T96 is worth doing**, but not urgently. The current T89c visualisation is
physically reasonable for storm-level exploration. T96 becomes clearly worthwhile once:
(a) real historical solar wind data drives the model (Phase 4), or
(b) users need accurate event reproductions. Port T96 first, defer T01 to Phase 4.
